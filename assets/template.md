# 报告模板

本模板用于指导 AI 直接基于 `merge` 与 `enrich` 后的数据生成最终 Markdown 报告。

这里定义的是报告结构、写作要求和补全边界，不是固定措辞。实际出稿时，应根据当天信息密度调整篇幅，不要机械填满每个小节。

## Markdown 兼容要求

为确保最终内容通过 `scripts/sanitize-html.py` 转换为 HTML 时不失真，最终成稿应只使用以下 Markdown 写法：

- 标题：`#`、`##`、`###`、`####`
- 普通段落
- 单层无序列表：`- `
- 标准表格：使用 `|` 分隔列，并单独保留表头分隔行
- 行内强调：`**加粗**`
- 行内代码：`` `code` ``
- 标准链接：`[标题](https://example.com)`
- 分隔线：`---`

请避免使用以下写法：

- 嵌套列表
- 有序列表作为正文主体结构
- 原始 HTML 标签
- 任务列表
- 脚注
- 表格单元格内换行或复杂排版
- 在表格前后不留空行

## 输入要求

生成报告前，优先使用已经完成选择性富化的合并结果。

对每条高价值新闻，尽量保留或整理出以下字段：
- 标题
- 来源
- 发布时间
- 原文链接
- 主题或分类
- 摘要或抓取片段
- 富化后的正文内容
- `quality_score`，仅作为排序参考，不作为最终结论

如果标题或来源标签明显不完整，可以依据正文、链接或上下文做保守规范化处理。

时间展示要求：
- 面向用户展示的日期与时间统一使用北京时间 `UTC+8`
- 日期格式统一为 `YYYY-MM-DD`
- 如果需要展示具体时间，使用 `YYYY-MM-DD HH:MM`

不要编造以下信息：
- 缺失的发布日期
- 原文未明确出现的机构名称
- 缺乏文本依据的政策意图或影响判断

如果信息不完整，请明确标注，例如：`信息不完整，需复核原文`。

## 写作规则

- 优先使用正式、完整的官方机构名称，不要直接沿用过短或噪声较多的 feed 标签。
- 点评要克制、基于来源，不要为了“有观点”而延伸推断。
- 对例行通知、常规会议、低增量信息进行压缩处理，不要平均分配篇幅。
- 如果多篇文章描述的是同一发展，应合并成一条叙事，并优先保留最强、最原始的来源链接。
- 如果富化后发现原标题过于笼统、误导或残缺，可以改写成更清晰的规范化标题，但必须忠于原文事实。
- 如果当天没有明显高价值进展，应直接说明“当日强信号有限”，并缩短报告篇幅。
- 如无法确认日期、来源、政策层级或影响范围，应明确写出不确定性，不要猜测补齐。
- 面向用户展示的所有日期和时间，默认按北京时间 `UTC+8` 书写；日期必须使用 `YYYY-MM-DD`。
