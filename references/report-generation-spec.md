# AI 出稿执行规范

本文件用于约束 AI 如何基于 `merge` 与 `enrich` 后的数据生成最终日报。

目标不是“写得像报告”，而是“稳定、可执行、可复核地生成报告”。当模板、风格和输入发生冲突时，以本规范为准；当本规范与事实材料冲突时，以事实材料为准。

## 1. 任务定义

AI 的默认任务是：

1. 读取 `merge` 后的候选集，以及必要时的 `enrich` 结果。
2. 判断哪些内容值得进入最终报告。
3. 对可疑、不完整或噪声较大的标题和来源进行保守规范化。
4. 用中文生成最终 Markdown 报告。
5. 明确保留不确定项，不得用猜测填补事实空白。

默认输出应遵循 `assets/template.md` 的结构。

## 2. 时间与日期规则

这是硬性要求，必须执行：

- 面向用户展示的所有日期和时间统一使用北京时间 `UTC+8`
- 日期格式统一为 `YYYY-MM-DD`
- 如需展示具体时刻，格式统一为 `YYYY-MM-DD HH:MM`
- 不要在正文里混用其他时区写法
- 如果原始数据是其他时区，应先换算为北京时间再展示
- 如果原文没有明确时间，只能写“原文未明确标注”，不得补写具体日期

示例：
- 正确：`2026-03-30`
- 正确：`2026-03-30 14:30`
- 错误：`2026/03/30`
- 错误：`30-03-2026`
- 错误：`Mar 30, 2026`

## 3. 输入字段约定

AI 应优先识别以下字段；字段名可不完全一致，但语义应尽量映射到这些槽位：

- `title`: 原始标题
- `source`: 来源名称或站点名称
- `link`: 原文链接
- `published`: 发布时间
- `summary`: 抓取摘要、snippet 或简要内容
- `content`: 合并结果里的正文片段
- `full_text`: 富化后的正文全文或长文本
- `topic`: 主题分类
- `quality_score`: 质量分，仅作参考
- `source_type`: RSS、Web 等来源类型
- `editorial_note` / `commentary` / `analysis`: 已有说明性字段，如存在可参考但不可盲从

字段优先级建议：

1. `full_text`
2. `content`
3. `summary`
4. `title`

也就是说，凡是需要补全、纠偏、点评时，应优先依据正文，再退回摘要和标题。

## 3.1 Markdown 兼容规则

为确保生成的 Markdown 经 `scripts/sanitize-html.py` 转换为 HTML 时尽量不失真，最终成稿只应使用以下语法：

- `#`、`##`、`###`、`####` 标题
- 普通段落
- 单层无序列表，列表项使用 `- `
- 标准 Markdown 表格
- `**加粗**`
- `` `行内代码` ``
- `[文字](https://example.com)` 链接
- `---` 分隔线

禁止在最终成稿中使用以下内容：

- 嵌套列表
- 表格单元格中的多段落内容
- 原始 HTML
- 任务列表
- 脚注
- 非标准 Markdown 扩展语法

表格要求：

- 表格前后留空行
- 每一行列数保持一致
- 不要在单元格内放置复杂列表
- 不要保留未替换的模板占位符

## 4. 选题规则

优先保留以下内容：

- 新发布的政策文件、征求意见稿、实施细则、正式通知
- 明确体现监管方向、执法尺度、风险提示的监管信息
- 对区域经济、金融活动、产业支持有实际意义的项目动态
- 具有明显增量信息的会议结论、数据发布、执行安排

通常压缩或降权以下内容：

- 例行会议新闻
- 无明确政策增量的宣传稿
- 重复转载
- 标题看起来重要，但正文缺乏实质信息的稿件
- 与金融政策或政府治理关系较弱的外围内容

## 5. 标题补全规则

允许对标题做“规范化”，但不允许做“创作性改写”。

允许的情况：

- 标题明显残缺
- 标题只有栏目名或通用词，无法看出事件主体
- 标题省略了关键机构、文件动作或对象范围，而正文中可以明确补足
- 多来源合并后，需要一个统一标题来代表同一事件

补全原则：

- 优先保留原文核心动词和政策动作
- 补出机构、对象、事项范围，但不要加入正文没有的结论
- 如果原题是正式文件标题，优先保留正式标题，不要擅自简化
- 如果只能部分判断，应少补，不要补满

示例：

- 原标题：`通知`
  可规范化为：`深圳市有关部门发布项目申报通知`
  前提：正文明确写明发布主体与事项

- 原标题：`关于做好有关工作的公告`
  可规范化为：`广东有关部门发布金融支持相关工作公告`
  前提：正文明确指向“金融支持相关工作”

以下情况禁止改写：

- 仅凭链接路径猜完整政策标题
- 仅凭主题分类推断具体政策内容
- 把“征求意见稿”写成“正式发布”
- 把“会议强调”写成“出台新规”

## 6. 来源补全规则

允许规范化来源名称，但必须基于已有证据。

允许的情况：

- feed 名称过短，例如只写“官网”“新闻”“通知公告”
- 来源名称含噪声、乱码、截断
- 多个来源其实指向同一官方机构，需要统一写法

规范化原则：

- 优先使用正式机构名或站点名
- 能精确到“人民银行”“国家金融监督管理总局”时，不要只写“监管部门”
- 如果只能确认站点域名，允许用“某机构官网”式保守表达
- 不要把二级栏目名误写成机构全称

示例：

- `pbc.gov.cn` 可规范化为 `中国人民银行`
- `nfra.gov.cn` 可规范化为 `国家金融监督管理总局`
- 若仅能确认 `beijing.gov.cn` 某栏目，可写 `北京市政府官网`

禁止：

- 凭经验把不明确来源写成具体司局
- 把转载媒体误写成政策发布机关

## 7. 摘要生成规则

每条重点新闻的摘要应回答：

- 谁发布了什么
- 针对什么对象或事项
- 当前处于什么阶段
- 为什么进入当天报告

摘要要求：

- 2-4 句为宜
- 优先写增量信息，不重复空泛标题
- 不堆砌原文套话
- 不写超出材料支持范围的因果判断

如果信息不足以形成完整摘要，应保守缩短，并标注信息不足。

## 8. 点评风格规则

点评是“编辑判断”，不是“观点输出”。

点评应聚焦以下一种或两种：

- 政策意义
- 监管信号
- 执行层面的实际影响
- 后续值得观察的点

点评要求：

- 每条 1-2 句
- 用词克制、明确、可落地
- 基于原文事实，不做宏大叙事
- 不写口号式判断，不写市场预测，不写政策必然效果

推荐表达：

- `这表明……`
- `这反映出……`
- `值得关注的是……`
- `后续可关注……`
- `从执行层面看……`

避免表达：

- `这将彻底改变……`
- `意义重大，影响深远`
- `预计将显著提振……`
- `说明监管将全面转向……`

## 9. 禁止猜测规则

以下内容一律禁止推断补齐：

- 未明确发布日期时，补写具体日期
- 未明确发文机关时，补写具体部门名称
- 未明确政策效力时，补写为“正式文件”“实施细则”“监管新规”
- 未明确金额、范围、期限时，补写数字或适用对象
- 未明确影响时，写成确定性经济或市场结论

如材料不足，只能使用以下表达：

- `原文未明确标注`
- `从现有文本无法确认`
- `信息不完整，需复核原文`
- `目前更像是执行信号，尚不足以判断为新增政策`

## 10. 去重与合并叙事规则

当多条内容实为同一事件时，应合并为一条叙事。

合并依据：

- 标题高度相似
- 链接指向同一政策或同一事件不同稿件
- 正文主体、时间、事项范围一致

合并后处理方式：

- 选最强原始来源作为主链接
- 统一成一个规范化标题
- 摘要中说明核心事实
- 点评只写一次

不要把同一事件拆成多条“看起来很多”的内容。

## 11. 数据概览规则

如果模板包含“数据概览”模块，应将其视为正式输出的一部分，而不是草稿占位区。

执行要求：

- 不得在最终报告中保留 `{{...}}` 形式的占位符
- 必须将占位符替换为真实统计值
- 统计值应与正文中实际纳入报告的内容口径一致
- 如果某一列不适用，可使用 `-`，但不要伪造数字
- 总计行必须与上方分项可核对

优先统计维度：

- 政策文件
- 项目动态
- 监管通知
- 合计

如果当天某些机构或区域没有可纳入内容，可写 `0` 或 `-`，但要保持整张表口径一致。

## 12. 输出质量要求

最终输出必须满足：

- 中文撰写
- Markdown 格式
- 结构遵循 `assets/template.md`
- 日期展示使用北京时间 `UTC+8`
- 日期格式使用 `YYYY-MM-DD`
- 链接保留可访问原文 URL
- 重点与例行内容有明显区分
- 对不确定项有单独说明
- 如果模板中包含“数据概览”模块，应按模板输出，并确保统计口径与正文一致

如果当天强信号有限，应明确写出：

- `当日强信号有限`
- `以例行更新为主`

不要为了“看起来充实”而强行拔高普通内容。

## 13. 推荐执行顺序

建议 AI 在出稿时按以下顺序思考：

1. 先看哪些条目最值得进“核心关注”
2. 再处理标题、来源、日期的规范化
3. 再写“今日主线”
4. 再补“分主题简报”
5. 再整理“数据概览”
6. 最后列“风险与不确定项”

这样可以减少先入为主，也能降低把低价值内容写重的风险。
